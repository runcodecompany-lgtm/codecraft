نظرة عامة على المشروع

Code Craft Core منصة تعليمية عربية لتعلّم البرمجة تعتمد على التلعيب (Gamification) ونظام اقتصادي داخلي (نقاط XP وعملات Craft Coins) لتحفيز المتعلمين.

الرؤية

توفير رحلة تعلّم برمجي متكاملة باللغة العربية، من المستوى المبتدئ إلى الاحتراف، عبر مسارات منظّمة، ودروس فيديو ونصية، واختبارات وتحديات وألعاب برمجية، مع نظام معلمين لإنشاء وإدارة المحتوى ولوحة إدارة مركزية.

الوحدات الأساسية للنظام

التعلّم والمحتوى

مسارات تعلّم (Tracks)، مقررات، وحدات، دروس، مصادر، ومقالات مع محرر نصوص غني.

التقييم والاختبارات

اختبارات (Quizzes)، بنك أسئلة، محاولات، واختبارات تحديد المستوى المدعومة بالذكاء الاصطناعي.

التلعيب والاقتصاد

نقاط الخبرة XP، عملات Craft Coins، الشارات، الإنجازات، لوحة الصدارة، والسلاسل اليومية (Streaks).

منظومة المعلمين

طلبات الانضمام، ملفات المعلمين، سير الموافقات (Approvals)، والمحافظ والمدفوعات.

المجتمع والتفاعل

منتديات، أسئلة وأجوبة، تعليقات، تفاعلات، متابعة، رسائل خاصة، وبلاغات.

الذكاء الاصطناعي

مساعد ذكي، توصيات، بطاقات تعليمية، توليد اختبارات، تلخيص، وتحليلات تعلّم.

تحقيق الدخل

اشتراكات، مدفوعات، فواتير، مشتريات، كوبونات، عروض، وبرنامج تسويق بالعمولة.

الإدارة والأمان

نظام صلاحيات RBAC، سجلّات تدقيق، إعدادات النظام، وتحديد معدّل الطلبات.

تحتوي قاعدة البيانات على أكثر من 80 نموذجاً (Model) موزّعة على هذه الوحدات، ما يعكس اتساع نطاق المنصة وتكاملها.

معمارية النظام

يعتمد المشروع على معمارية موحّدة قائمة على Next.js App Router تدمج الواجهة والخادم في تطبيق واحد (Full-Stack) مع فصل واضح للمسؤوليات عبر الطبقات.

معمارية النظام

تفصيل طبقات النظام ومسؤولية كل طبقة.

1) طبقة العرض (Presentation Layer)

مكوّنات React (Server & Client) في app/ وcomponents/ — الصفحات، لوحات التحكم (طالب/معلم/مدير)، والنماذج التفاعلية.

2) طبقة الحافة (Edge / Middleware)

middleware.ts يتحقق من جلسة Supabase، ويحرس مسارات /dashboard، ويلتقط كود الإحالة، ويعيد توجيه المستخدمين المصادَق عليهم.

3) طبقة التطبيق (Application Layer)

actions/* (Server Actions) وapp/api/* (API Routes) — نقاط الدخول لمنطق الأعمال، محميّة عبر guardAction و RBAC ومحدّد معدّل الطلبات.

4) طبقة منطق الأعمال (Domain / Services)

lib/* — خدمات مثل المصادقة، التلعيب، التعلّم، الذكاء الاصطناعي، الحوكمة والهوية، والأساس (foundation).

5) طبقة البيانات (Data Layer)

Prisma ORM فوق PostgreSQL، إضافة إلى Supabase (المصادقة والتخزين) ومزوّدي الذكاء الاصطناعي (Gemini / Groq).

مبادئ التصميم المعتمدة

  • فصل المسؤوليات: كل طبقة معزولة ولا تتجاوز حدودها؛ الواجهة لا تصل للبيانات مباشرة بل عبر Server Actions.
  • الأمان أولاً بالتصميم: كل عملية حسّاسة تمرّ عبر guardAction للتحقق من الجلسة والصلاحية والحد من التكرار.
  • الذرّية (Atomicity): العمليات المركّبة (كالتسجيل) تُنفّذ داخل معاملات Prisma ($transaction).
  • قابلية التوسّع: فصل مزوّدي الخدمات (Supabase / AI / S3) خلف وحدات مستقلة يسهّل استبدالها.
  • اختلاف البيئات: إعدادات أكثر صرامة في الإنتاج (فحص الأنواع، رؤوس الأمان، الكاش، تحديد المعدّل).

طبقة العميل وطبقة الخادم

يفصل النظام بوضوح بين ما يُنفَّذ في متصفح المستخدم (Client) وما يُنفَّذ على الخادم (Server) لضمان الأمان والأداء.

طبقة العميل (Client Side)

تعمل في المتصفح وتُعنى بالتفاعل والعرض فقط:

  • Client Components ("use client")
  • النماذج التفاعلية عبر react-hook-form + Zod
  • محرّرات Monaco و Tiptap والألعاب البرمجية
  • إدارة الحالة والثيم (next-themes)
  • توليد الشهادات PDF (jspdf/html2canvas)

طبقة الخادم (Server Side)

تعمل على الخادم وتحتكر منطق الأعمال والبيانات:

  • Server Components لجلب البيانات مباشرة
  • Server Actions ("use server") للعمليات
  • API Routes للتكاملات وفحوص الصحة
  • Prisma للوصول لقاعدة البيانات
  • المصادقة، RBAC، والأسرار (Secrets)

شكل 7-1: مخطّط حالات الاستخدام مع علاقات «include» و«extend».

علاقة «include» (تضمين)

سلوك إجباري تشترك فيه عدّة حالات. مثال: التسجيل، المشاهدة، وحل الاختبار جميعها «تتضمّن» التحقق من الهوية؛ ولا يكتمل «التسجيل في مقرر» مدفوع دون «الدفع».

علاقة «extend» (توسعة)

سلوك اختياري يُضاف عند تحقّق شرط. مثال: إصدار الشهادة يوسّع حل الاختبار (عند النجاح)، ومنح XP يوسّع المشاهدة/الاختبار (عند الإكمال).

المصادقة وإدارة الجلسات

  • Supabase Auth لإدارة الحسابات والجلسات، إلى جانب رموز جلسات محلية (session_token) يتم التحقق منها عبر validateSession.
  • تجزئة كلمات المرور باستخدام bcrypt (Salt Rounds) قبل التخزين — لا تُخزَّن كلمات المرور نصّياً أبداً.
  • التحقق من البريد الإلكتروني عبر رموز (Tokens) مجزّأة ومحدودة الصلاحية (EmailVerification, PasswordReset).
  • حراسة المسارات في middleware.ts: منع الوصول غير المصادَق عليه إلى /dashboard وإعادة التوجيه.

الصلاحيات والتحكم بالوصول (RBAC)

  • نظام أدوار متدرّج: GUESTSTUDENTTEACHERMODERATORADMINSUPER_ADMIN
  • جداول Role / Permission / RolePermission لربط الصلاحيات بالأدوار ديناميكياً.
  • الدوال hasPermission, hasRole, وcheckPermissionOrThrow تفرض التحقق قبل كل عملية حسّاسة.
  • guardAction يجمع: التحقق من الجلسة + الدور + تأكيد البريد + اعتماد المعلّم + التبريد الزمني (Cooldown).

حماية المدخلات والطلبات

  • التحقق بـ Zod لكل المدخلات قبل معالجتها لمنع البيانات غير الصالحة.
  • Prisma ORM يستخدم استعلامات مُعامَلة (Parameterized) تمنع حقن SQL بطبيعتها.
  • تحديد معدّل الطلبات (Rate Limiting) عبر rate-limiter.ts لصدّ الهجمات التكرارية (يعيد 429 عند التجاوز).

أمن البيانات وسلامتها ونزاهتها

تابع — رؤوس الأمان، سلامة البيانات، والنزاهة والتدقيق.

رؤوس الأمان (Security Headers)

تُطبَّق على مستوى next.config.ts لكل المسارات:

الرأسالقيمة / الغرض
X-Frame-OptionsDENY — منع التضمين داخل إطارات (Clickjacking).
X-Content-Type-Optionsnosniff — منع تخمين نوع المحتوى.
Strict-Transport-Securityفرض HTTPS لمدة سنة مع النطاقات الفرعية.
Referrer-Policystrict-origin-when-cross-origin.
Permissions-Policyتعطيل الكاميرا/الميكروفون/الموقع وتتبّع FLoC.
X-XSS-Protectionحماية إضافية ضد هجمات XSS.

سلامة البيانات (Data Integrity)

  • قيود قاعدة البيانات: مفاتيح أساسية (UUID)، قيود التفرّد (@unique على البريد وكود الإحالة ورقم الشهادة)، ومفاتيح مركّبة فريدة (@@unique([userId, courseId])).
  • المفاتيح الأجنبية وسياسات الحذف: onDelete: Cascade للحفاظ على الاتساق، وSetNull حيث يلزم منع فقدان السجلّات.
  • الفهارس (Indexes): على الحقول كثيرة الاستعلام (البريد، الدور، الحالة) لضمان الأداء والاتساق.
  • المعاملات الذرّية: استخدام prisma.$transaction في العمليات المركّبة (التسجيل، إنشاء طلب المعلم) لضمان مبدأ «الكل أو لا شيء».
  • القيم المُعدّدة (Enums): تقييد الحقول لقيم صحيحة فقط (الأدوار، الحالات، مستويات الصعوبة).

النزاهة والتدقيق (Auditability)

  • سجلّ التدقيق (AuditLog): تسجيل الأحداث الحسّاسة (تسجيل، تعديل صلاحيات) مع عنوان IP ووكيل المستخدم.
  • سجلّات المعاملات الاقتصادية (Transaction, XpTransaction) تُنشئ أثراً غير قابل للتلاعب لكل تغيّر في النقاط والعملات.
  • فحص الصحة عبر /api/health لمراقبة حالة الخادم وقاعدة البيانات، مع جاهزية للربط بـ Sentry.
خلاصة: يجمع النظام بين المصادقة القوية، والصلاحيات الدقيقة، والتحقق الصارم من المدخلات، وقيود قاعدة البيانات، والمعاملات الذرّية، وسجلّات التدقيق — لتحقيق السرّية والسلامة والتوافر (CIA Triad).
