# Code Craft Core - Database, Security, & Testing Skill 🔒

تحدد هذه المهارة المعايير والممارسات اللازمة للتعامل مع قاعدة البيانات، وضمان أمان المنصة وحماية بيانات المستخدمين، بالإضافة إلى معايير كتابة اختبارات الوحدة.

---

## 🗄️ 1. تصميم قواعد البيانات وإدارة Prisma
- **سلامة العلاقات والـ Schemas**: عند التعديل على ملف `prisma/schema.prisma` لتعريف جداول أو حقول جديدة، تأكد من:
  - ربط الحقول بشكل صحيح باستخدام علاقات (`@relation`).
  - تحديد شروط الحذف والتعديل التلقائي (`onDelete: Cascade` أو `Restrict` بحسب تدفق العمل).
  - كتابة تعليقات توضيحية أمام الحقول الحساسة.
- **استخدام المعاملات البرمجية (Transactions)**: عند إتمام عمليات كتابة متعددة في وقت واحد (مثل خصم نقاط وتحديث شعلة التعلم وإضافة سجل إنجاز)، استخدم دائماً `db.$transaction` لضمان التراجع عن الخطوات في حال فشل أي جزء منها:
```typescript
const result = await db.$transaction(async (tx) => {
  // 1. خصم العملات
  await tx.userProfile.update({ ... });
  // 2. تسجيل شراء العنصر
  await tx.purchase.create({ ... });
  
  return { success: true };
});
```

## 🔐 2. الأمن وإدارة الصلاحيات (Authentication & RBAC)
- **التحقق من الصلاحيات**: لا تعتمد فقط على حماية المسارات في الـ Middleware. يجب التحقق من هوية المستخدم ودوره داخل كل Server Action و API Route كخط دفاع إضافي:
  - **الأدوار المعتمدة**: `STUDENT` (طالب)، `TEACHER` (معلم)، `ADMIN` (مدير)، `SUPER_ADMIN` (مدير عام).
- **التحقق من الملكية (Ownership Verification)**: تأكد من أن المعلم لا يمكنه تعديل أو حذف دورة تعليمية إلا إذا كان هو من قام بإنشائها، أو كان مديراً للمنصة.
- **حماية البيانات الحساسة**: تجنب إرجاع كلمات المرور أو حقول التوكنات أو البيانات الحساسة من استعلامات قاعدة البيانات إلى الواجهة الأمامية (استخدم كتل `select` لاستثناء الحقول السرية).

---

## 🛡️ 3. التحقق الصارم من المدخلات (Zod Validation)
يجب التحقق من أي بيانات تأتي من المستخدمين أو الأنظمة الخارجية بشكل كامل:
- عرّف Schemas ثابتة وموثقة لكل طلب مستخدم.
- استخدم رسائل خطأ واضحة باللغة العربية لعرضها للمستخدم عند فشل التحقق من البيانات.
```typescript
const profileSchema = z.object({
  fullName: z.string().min(3, "الاسم يجب أن لا يقل عن 3 أحرف").max(50),
  bio: z.string().max(200, "نبذة التعريف يجب أن لا تتجاوز 200 حرف").optional(),
});
```

## 🧪 4. معايير الاختبارات (Unit Testing with Vitest)
- **أين تكتب الاختبارات**: يجب وضع ملفات الاختبار بجانب الكود البرمجي الذي تختبره، باستخدام اللاحقة `.test.ts` أو `.spec.ts`.
- **محاكاة قاعدة البيانات (Mocking)**: عند اختبار الـ Server Actions أو الخدمات التي تعتمد على Prisma، استخدم أدوات المحاكاة لمنع الكتابة الفعلية في قاعدة البيانات التجريبية.
- **تشغيل الاختبارات**: تأكد دائماً من تشغيل الاختبارات المحلية باستخدام الأمر:
  ```bash
  npx vitest run
  ```
  وتأكد من أن كودك الجديد لا يتسبب في فشل أي اختبارات سابقة.
